## P-4.1. 다크 모드 변형 (전체 화면)

**[목적]**
23개 화면의 다크 모드 버전을 만들어, 라이트/다크 토큰 매핑이 일관되는지 시각적으로 검증한다.

**[참조 문서]**
- `04_와이어프레임.md` §1.2 (다크 컬럼), §7.1 (다크 모드)

**[전제 조건]**
- 단계 3 완료. `03_screens_auth.pen`, `04_screens_main.pen` 화면 모두 default 상태로 존재.
- 새 캔버스 `design/05_states.pen` 또는 기존 캔버스에 우측 컬럼으로 추가.

**[프롬프트]**
```
[공통 시작 지시문 적용]

지금까지 만든 23개 화면(S01~S23)의 다크 모드 변형을 만들어줘.

방법:
1) 기존 라이트 모드 프레임 우측에 같은 크기의 다크 모드 프레임을 복제
2) 컬러 토큰을 라이트→다크 매핑으로 자동 치환:
   - --color-bg: #FFFAF5 → #1F1B1A
   - --color-surface: #FFFFFF → #2A2625
   - --color-text-primary: #2A2825 → #F5F1EC
   - --color-text-secondary: #7A736D → #A8A097
   - --color-divider: #EFE8E0 → #3A332E
   - --color-primary: #FF7E6B → #FF9580
   - --color-primary-soft: #FFE8E2 → #3A2A26
   - --color-secondary: #7EB6E8 → #9BC8F0
   - 그 외 success/warning/error/overlay는 04_와이어프레임.md §1.2 표 그대로

3) 각 다크 프레임 좌측 상단에 라벨 "S##-XXX (Dark)" 추가

4) 우선순위 5개 화면(S12 홈, S14 채팅, S16 지도, S18 캘린더, S20 학원)은 특히 신경써서:
   - 사진/이미지 영역은 디머가 아닌 의도적 명도 보정 적용
   - 가족 컬러 8종은 다크 배경에서도 가독성 유지하는 톤으로 (P-1.2 검증 결과 반영)

5) 정렬: 라이트 프레임과 다크 프레임을 가로로 짝지어 배치, 그 짝들을 세로로 쌓음
```

**[제약 조건]**
- 가족 컬러는 라이트/다크 둘 다 같은 의미이지만 명도는 다크에서 약간 밝게 (P-1.1에서 정의한 다크 가족 컬러 사용)
- WCAG AA 4.5:1 유지

**[성공 기준]**
- [ ] 23개 다크 변형 모두 생성
- [ ] 라이트/다크 짝지어 배치
- [ ] 다크에서도 컴포넌트 인스턴스 사용 (변형 전환만)

**[다음 단계]** → P-4.2

## P-4.2. 로딩 상태 (Skeleton)

**[목적]**
주요 화면의 데이터 로딩 중 상태를 Skeleton 형태로 만든다.

**[참조 문서]**
- `04_와이어프레임.md` §6 (상태 패턴), 각 화면의 `loading` 상태

**[전제 조건]**
- 단계 3 완료
- 우선순위 화면 위주

**[프롬프트]**
```
[공통 시작 지시문 적용]

다음 7개 화면의 "loading (Skeleton)" 상태를 변형으로 만들어줘. 각 화면 옆/아래에 새 프레임으로 추가.

대상 화면:
- S12 홈
- S13 채팅 목록
- S14 채팅방
- S16 지도
- S18 캘린더
- S20 학원 관리(목록)
- S21 알림 인박스

Skeleton 규칙:
- 모든 텍스트/이미지/카드를 회색 둥근 사각형으로 치환
- 컬러: --color-divider (라이트), 다크는 매핑
- 모서리: 원본과 동일 (텍스트는 4, 카드는 16)
- 텍스트는 70% 폭의 막대로 (한 줄), 본문은 100% 폭 + 마지막 줄 60%
- 부드러운 펄스 애니메이션 가이드 — 문서엔 "pulse 1.5s ease-in-out infinite" 메모

화면별 특이사항:
- S14 채팅방: 메시지 버블 모양은 유지, 안의 텍스트만 회색 막대
- S16 지도: 지도 영역은 회색 그라데이션, 핀은 회색 원
- S18 캘린더: 그리드는 그대로 두고, 일정 도트와 시트 카드만 회색
- S12 홈: 4개 카드 모두 Skeleton

각 프레임 라벨 "S##-XXX (Loading)" 추가.
```

**[성공 기준]**
- [ ] 7개 화면 Skeleton
- [ ] 컴포넌트 구조 유지, 콘텐츠만 회색
- [ ] 텍스트 막대 70%/100%/60% 룰 일관

**[다음 단계]** → P-4.3

## P-4.3. 빈 상태 (Empty)

**[목적]**
데이터가 0건일 때의 화면을 EmptyState 컴포넌트(P-2.7)를 활용해 표시.

**[참조 문서]**
- `04_와이어프레임.md` §5.3, 각 화면의 `empty` 상태

**[프롬프트]**
```
[공통 시작 지시문 적용]

다음 6개 화면의 "empty" 상태 변형을 만들어줘. EmptyState 컴포넌트 인스턴스를 사용.

대상:
1) S12 홈 — 가족 0명 (본인만): EmptyState "가족 없음" 변형 사용 + [초대하기]
2) S13 채팅 목록 — 메시지 0건: "아직 채팅이 없어요. 가족에게 첫 메시지를 보내보세요" + [채팅 시작]
3) S14 채팅방 — 메시지 0건: "첫 메시지를 보낼까요?" + 추천 칩 3개 ("좋은 아침!", "지금 어디?", "사랑해 ❤")
4) S18 캘린더 — 선택일 일정 0개: 시트에 EmptyState "일정 없음" + [일정 추가]
5) S20 학원 관리 — 학원 0개: "학원을 등록하면 캘린더에 자동 표시돼요" + [학원 등록]
6) S21 알림 인박스 — 알림 0건: "받은 알림이 없어요" (CTA 없음, 일러스트만)

화면 구조 유지 (헤더, 탭바 등) + 본문만 EmptyState로 채움.

각 프레임 라벨 "S##-XXX (Empty)".
```

**[성공 기준]**
- [ ] 6개 화면 empty 상태
- [ ] EmptyState 컴포넌트 인스턴스 사용 (P-2.7)
- [ ] 헤더/탭바는 default와 동일

**[다음 단계]** → P-4.4

## P-4.4. 에러 상태 (Error / Network / Server)

**[목적]**
네트워크 오류, 서버 오류, 권한 만료 등의 에러 상태를 표준 패턴으로.

**[참조 문서]**
- `04_와이어프레임.md` §6 (전역 상태 패턴: error-network, error-server)

**[프롬프트]**
```
[공통 시작 지시문 적용]

다음 4가지 에러 상태 패턴을 표준화해서 화면에 적용해줘.

A) 네트워크 오류 (offline) — 화면 상단 노란 띠
   - 높이 32, 배경 --color-warning, 텍스트 white
   - 메시지 "오프라인 — 캐시된 정보를 보여드려요"
   - 상태 화면: S12 홈 위에 띠가 추가된 변형 (S12-OFFLINE)

B) 전송 실패 — 메시지 옆 ⚠ 표시
   - S14 채팅방 변형: 본인 메시지 1건이 sending 실패한 상태
   - 메시지 우하단에 ⚠ 노란 아이콘 + 작은 텍스트 "전송 실패" + Ghost 버튼 [다시 보내기]

C) 서버 오류 (5xx) — 페이지 중앙 메시지
   - S13 채팅 목록 변형: 본문 영역에 EmptyState 변형
   - 일러스트 placeholder + "잠시 후 다시 시도해 주세요" + Button.Secondary [재시도]
   - (S13-ERROR-SERVER)

D) 인증 만료 — 모달
   - S12 홈 위에 ModalDialog (info-only 변형)
   - "다시 로그인이 필요해요" + "보안을 위해 로그아웃됐어요. 다시 로그인해 주세요." + [확인]
   - (S12-AUTH-EXPIRED)

각 프레임 라벨에 상태 명시 (S##-XXX (Error: type)).
```

**[성공 기준]**
- [ ] 4 에러 패턴 모두 표시
- [ ] 토스트가 아닌 인라인 패턴 위주 (지속적 상태)
- [ ] 재시도/확인 액션 모두 존재

**[다음 단계]** → P-4.5

## P-4.5. 권한 거부 상태 (Permission Denied)

**[목적]**
OS 권한(위치, 푸시 알림, 카메라, 사진)이 거부된 상태를 표시.

**[참조 문서]**
- `04_와이어프레임.md` §6 (permission-denied), 각 화면의 권한 상태 명세

**[프롬프트]**
```
[공통 시작 지시문 적용]

3가지 권한 거부 상태를 만들어줘.

A) 위치 권한 거부 — S16 지도
   - 지도 영역 위에 EmptyState 오버레이:
     - 일러스트: 지도 핀 + 자물쇠 아이콘
     - "위치 권한이 필요해요"
     - body secondary: "가족 위치를 보려면 OS 위치 권한을 켜주세요. 위치 정보는 가족 외부로 절대 공유되지 않아요."
     - Button.Primary [설정 열기] (시스템 deeplink)
   - 하단 BottomSheet도 회색 처리 (탭은 보이지만 데이터는 dim)
   - 라벨 "S16-PERMISSION-LOCATION"

B) 푸시 알림 권한 거부 — S21 알림 인박스
   - 인박스 상단에 인라인 배너 (높이 48):
     - 배경 --color-warning-soft (warning 30%), 보더 --color-warning 1px
     - 텍스트 "푸시 알림이 OS에서 꺼져 있어요" + 우측 [켜기] Button.Ghost
   - 그 아래 정상 알림 인박스
   - 라벨 "S21-PERMISSION-PUSH"

C) 카메라/사진 권한 거부 — S15 첨부 시트
   - "사진 보내기"를 탭했는데 권한이 거부된 경우
   - 작은 모달:
     - "사진 권한이 필요해요"
     - "프로필 사진을 바꾸거나 채팅에 사진을 보내려면 사진 접근 권한을 켜주세요."
     - [설정 열기] [취소]
   - 라벨 "S15-PERMISSION-PHOTO"

각 권한 거부 패턴은 일관:
- 친근한 톤 ("필요해요", "켜주세요")
- 이유 설명 (왜 필요한지)
- 시스템 설정 deeplink CTA
```

**[성공 기준]**
- [ ] 3 권한 패턴 모두 표시
- [ ] 친근한 톤 일관
- [ ] [설정 열기] CTA 모두 포함

## 부록 — 변형 캔버스 정리 프롬프트

```
변형이 모두 만들어진 후 정리:

1) 같은 화면(S##-XXX)의 모든 변형(default/dark/loading/empty/error/permission)을 가로로 묶음
2) 각 줄 좌측에 화면 ID 라벨 + 변형 종류
3) Pencil의 Variants 기능으로 묶을 수 있으면 묶기 (S12-HOME 컴포넌트 + 변형 6개)
4) 우측 컬럼에 "구현 우선순위" 표시:
   - High: default + dark + loading
   - Medium: empty + error
   - Low: permission (있으면 좋지만 필수 아님)

이렇게 정리하면 개발자가 한 화면당 어떤 상태들을 구현해야 하는지 한눈에 보임.
```
